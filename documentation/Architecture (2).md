# `jerry` — Architecture Document

**Project:** jerry — Lawyer-Client Consultation Platform
**Version:** 1.0 (MVP)
**Last Updated:** April 2026
**Status:** Ready for Implementation

---

## 1. Executive Summary

`jerry` is a mobile-first marketplace connecting legal clients with verified lawyers across India for real-time consultation via **chat, voice call, and video call**. The platform supports four distinct roles (User, Lawyer, Admin, SuperAdmin) with a strict verification workflow for lawyers and a free-tier MVP that is designed to support monetization later without architectural rewrites.

The system is architected as a **client-heavy + thin-persistence backend**: chat history lives on the user's device (SQLite), the server stores only metadata and transient queues. This is cheap at scale and respects the low-compliance-burden brief (India testing phase, no long-term audit requirement).

## 4. Architectural Principles

1. **Stateless API Servers** — any NestJS instance can handle any request. State lives in Postgres + Redis only. Enables horizontal scaling.
2. **Client-Heavy Persistence** — chat history on device, not server. Reduces DB load, improves privacy, matches user's requirement.
3. **Separation of Identity Tables** — `users`, `lawyers`, `admins`, `super_admins` are distinct tables (not a single `users` table with a `role` column). Cleaner queries, clearer permissions, easier schema evolution.
4. **Abstract Third-Party Dependencies** — `CallService`, `PushService`, `StorageService` are interfaces. Agora/FCM/R2 are implementations. Swap providers in weeks, not months.
5. **Design-for-Payment Later** — `PaymentModule` exists in MVP as a stub. Schema fields (`rate_per_session`, `consultation.amount_paid`) exist but default to zero. Razorpay plug-in is a future PR.
6. **Encryption by Default** — TLS in transit, BYTEA license docs behind admin-only auth, JWT secrets in env, Postgres at-rest encryption via provider (RDS / DigitalOcean Managed PG).
7. **Offline-First Messaging** — every message route assumes receiver may be offline. FCM + `pending_messages` queue handles this cleanly.

### 5.3 Data Layer

**PostgreSQL 16** — primary data store
- Role tables (users, lawyers, admins, super_admins)
- Relational data (specialties, ratings, consultations)
- Transient queue (pending_messages)
- BYTEA column (license_document) on `lawyers` table

**Redis 7** — multi-purpose
- **Presence** — `presence:user:<id>` / `presence:lawyer:<id>` with TTL 30s
- **Socket.IO adapter** — enables multi-node Socket.IO clustering
- **OTP cache** — `otp:<email>` with TTL 10min (alternative to Postgres, faster)
- **Rate limiting** — per-IP and per-user token buckets
- **Session tokens** — refresh token allow-list (so revocation works)

**Cloudflare R2** — object storage
- `profile-photos/<user_id>.jpg` — public read, signed write
- Presigned POST URLs generated by MediaModule (5-min TTL)
- Client uploads directly to R2, bypasses NestJS bandwidth

## 7. Data Flow Architectures

### 7.1 Signup + Email OTP Flow
```
1. Flutter POST /auth/signup { email, password, fullName, role, language? }
2. AuthService:
   - Validate DTO (Joi / class-validator)
   - Check email doesn't exist across all 3 tables (users, lawyers, admins)
   - Generate 6-digit OTP
   - Store hash(otp) in Redis: otp:<email> = bcrypt(otp), TTL 10min
   - Call NotificationService.sendEmailOtp(email, otp) → Brevo API
   - Store pending signup data in Redis: pending_signup:<email> = {hashedPwd, fullName, role, language}, TTL 15min
   - Return 202 Accepted
3. Flutter POST /auth/verify-otp { email, otp }
4. AuthService:
   - Compare bcrypt(otp) vs stored hash
   - If match: fetch pending_signup, create row in users/lawyers table
   - Lawyer role: status=PENDING_LICENSE_UPLOAD (must upload license before they can log in)
   - User role: active immediately
   - Issue JWT (access 15min + refresh 30days)
   - Store refresh token hash in Redis: refresh:<userId>:<deviceId>
   - Return { accessToken, refreshToken, user }
```

### 7.2 Lawyer Approval Flow
```
1. Lawyer uploads license after signup:
   Flutter POST /license/upload (multipart) → LicenseModule
   - File saved as BYTEA on lawyers.license_document
   - lawyer.verification_status = PENDING_REVIEW
2. Admin Flutter GET /admin/approval-queue
   - Returns list of lawyers with status=PENDING_REVIEW
3. Admin GET /admin/lawyer/:id/license → returns BYTEA as binary stream
4. Admin POST /admin/lawyer/:id/approve OR /reject { reason? }
5. On approve:
   - lawyer.verification_status = APPROVED
   - lawyer.approved_by_admin_id = <currentAdminId>
   - NotificationService → FCM push to lawyer device: "Your account is approved, log in now"
6. On reject:
   - status = REJECTED, store verification_notes
   - FCM push with rejection reason
7. Login blocked for lawyers unless status = APPROVED
```

### 7.3 Chat Message Flow — Receiver Online
```
UserA taps send in Flutter
  → Flutter writes to local SQLite (status=SENDING)
  → socket.emit('chat:send', { toUserId, content, clientMessageId })
  ↓
ChatGateway receives event
  → validates JWT on socket
  → validates sender has active consultation or open thread with receiver
  → Redis GET presence:user:<toUserId>
  → if online: socket.to(receiverSocketId).emit('chat:receive', {...})
  → ack back to sender: 'chat:delivered' with server messageId
  ↓
Flutter (sender) updates SQLite status=DELIVERED
Flutter (receiver) appends to SQLite, emits 'chat:read' on screen-view
  → sender SQLite updates status=READ
```

### 7.4 Chat Message Flow — Receiver Offline
```
UserA sends message
  → ChatGateway → Redis GET presence:user:<toUserId> returns null
  → INSERT INTO pending_messages (recipient_id, sender_id, content, created_at)
  → NotificationService.push(receiverFcmToken, "New message from {senderName}")
  → ack to sender: 'chat:queued'
  ↓
UserB opens app (or taps notification)
  → Flutter socket.connect → ChatGateway: 'chat:sync'
  → SELECT * FROM pending_messages WHERE recipient_id = <userBId> ORDER BY created_at
  → emit each to socket → Flutter inserts to SQLite
  → on ack: DELETE FROM pending_messages WHERE id IN (...)
```

### 7.5 Video Call Flow
```
1. UserA on Lawyer profile → taps "Video Call"
2. POST /call/initiate { lawyerId, type: 'VIDEO' }
3. CallService:
   - Verify lawyer is_online=true AND not in_call
   - Create consultation row: {user_id, lawyer_id, type: VIDEO, status: RINGING, started_at: now()}
   - Generate Agora channel name: consult_<consultationId>
   - Generate Agora RTC tokens for both parties (valid 1hr)
   - Redis SET lawyer:<id>:in_call = consultationId (TTL 5s, refreshed by heartbeat)
   - Emit socket 'call:incoming' to lawyer's socket with { callerId, callerName, channelName, lawyerToken, consultationId }
   - Parallel: NotificationService.push(lawyerFcmToken, 'incoming_call', data-payload for Flutter to show call UI)
   - Return { channelName, userToken, consultationId } to caller
4. UserA Flutter joins Agora channel with userToken
5. Lawyer accepts:
   - POST /call/:consultationId/accept
   - consultation.status = ACTIVE
   - Flutter joins Agora channel with lawyerToken
   - Both parties now connected P2P via Agora SFU
6. Either party ends:
   - Flutter leaves Agora channel
   - POST /call/:consultationId/end
   - consultation.status = ENDED, ended_at=now(), duration_seconds=calculated
   - Redis DEL lawyer:<id>:in_call
   - After END: server emits 'rating:prompt' to user → rating flow
7. If lawyer doesn't accept within 45s:
   - Timeout job → consultation.status = MISSED
   - Emit 'call:missed' to user
```

### 7.6 Presence Tracking
```
On socket connect (after JWT validation):
  Redis SETEX presence:user:<id> 30 <socketId>
  Broadcast to interested parties (e.g., consultation participants): 'presence:online'

Every 20s heartbeat from Flutter:
  socket.emit('heartbeat')
  → ChatGateway → Redis SETEX presence:user:<id> 30 <socketId> (refresh TTL)

On socket disconnect:
  Redis DEL presence:user:<id>
  Broadcast 'presence:offline'

Lawyer list API (GET /lawyers?filter=...):
  Postgres query returns lawyers
  For each: Redis MGET presence:lawyer:<id1> presence:lawyer:<id2> ...
  Attach is_online boolean to each lawyer in response
```

### 7.7 Multi-Device Login Flow
```
1. User already logged in on Phone A:
   Redis has: session:user:<id>:devices = ["deviceA_id"]
   refresh:user:<id>:deviceA_id = <hashedRefreshToken>

2. User attempts login on Phone B:
   POST /auth/login { email, password, deviceId: deviceB_id, deviceInfo }
3. AuthService:
   - Validate credentials
   - Check Redis session:user:<id>:devices → finds deviceA
   - Return 409 Conflict: { existingDevice: { deviceInfo: 'iPhone 13 · Mumbai', lastActive: '2 min ago' } }
4. Flutter shows dialog: "Already logged in on {device}. Log out that device?"
5. User confirms:
   POST /auth/login { ..., forceLogout: true }
6. AuthService:
   - DEL refresh:user:<id>:deviceA_id (invalidates refresh token)
   - DEL session:user:<id>:devices
   - Send Socket.IO event to deviceA socket: 'auth:force_logout' → Flutter clears tokens, routes to login
   - Send FCM push to deviceA: "Logged out — signed in from another device"
   - Issue new tokens for deviceB_id
   - session:user:<id>:devices = ["deviceB_id"]
```

## 8. Database Architecture Principles

### 8.1 Separation of Identity Tables
- `users`, `lawyers`, `admins`, `super_admins` are **separate tables**.
- Each has its own email uniqueness constraint.
- Cross-table uniqueness enforced at AuthService layer via explicit checks.
- Benefit: fields differ significantly (lawyers have license, specialties, rates; users don't). Separate tables avoid nullable column explosion.

### 8.2 Indexing Strategy
- **Email columns** — unique B-tree index (login lookups)
- **`presence`-related** — none (presence is Redis-only)
- **`lawyers.verification_status`** — B-tree (admin approval queue)
- **`lawyers.city`, `lawyers.languages_spoken` (GIN)** — lawyer search
- **`lawyer_specialties (lawyer_id, specialty_id)`** — composite for join efficiency
- **`consultations (user_id, created_at DESC)`** — past-consults list
- **`consultations (lawyer_id, created_at DESC)`** — lawyer's history
- **`pending_messages (recipient_id, created_at)`** — offline message sync
- **`ratings (lawyer_id)`** + partial index on non-null text reviews

### 8.3 Connection Pooling
- Prisma's built-in connection pool (default 10 per instance)
- In production: **PgBouncer** in transaction mode (pool_mode=transaction)
- Per NestJS instance: max 10 connections to PgBouncer
- Enables 100+ API instances sharing a small Postgres connection pool

### 8.4 BYTEA vs Object Storage Split
| Content Type | Storage | Rationale |
|---|---|---|
| Lawyer license document | Postgres BYTEA | Sensitive, admin-only, viewed ~2x per lawyer lifetime (approval + audit) |
| Profile photos (users, lawyers) | Cloudflare R2 | Viewed 100s of times per session, CDN-cached, zero egress cost |
| Chat attachments (v2, not MVP) | Cloudflare R2 | High volume, public-read signed URLs |

## 9. Security Architecture

### 9.1 Authentication
- **JWT access token**: 15 min TTL, signed with RS256, carries `{sub, role, deviceId}`
- **JWT refresh token**: 30 days TTL, stored hashed in Redis (allow-list), single-use (rotated on each refresh)
- **Device-session enforcement**: single active device per user (latest login wins, with confirmation)
- **OTP verification**: 6-digit, bcrypt-hashed in Redis, 10-min TTL, 5-attempt lockout

### 9.2 Authorization (RBAC)
- `@Roles('USER', 'LAWYER')` decorator on controllers
- `RolesGuard` reads JWT claim, compares to route requirement
- **Admin endpoints** — only accessible with role=ADMIN or SUPER_ADMIN
- **SuperAdmin endpoints** — only role=SUPER_ADMIN (admin-creation, platform analytics)

### 9.3 Transport & Storage Security
- **TLS 1.3** everywhere (NGINX → API, Flutter → API, API → Postgres/Redis)
- **Password hashing** — bcrypt, cost factor 12
- **License BYTEA** — served only to role=ADMIN/SUPER_ADMIN via authenticated endpoint, never cached by Flutter
- **FCM tokens** — stored per-device, deleted on logout
- **Postgres at-rest** — handled by managed provider (e.g., DigitalOcean Managed PG AES-256)

### 9.4 Input Validation
- `class-validator` DTOs on every controller endpoint
- Socket.IO events: explicit schema validation before handler runs
- File uploads: MIME type whitelist + size cap (5MB for license PDF/JPG)

### 9.5 Rate Limiting
- Per-IP: 100 req/min on `/auth/*` endpoints
- Per-user: 60 req/min global
- Socket.IO: 20 chat messages/min per user (burst 5)
- Enforced via `@nestjs/throttler` + Redis backing store

### 9.6 Audit / Logging
- Structured logs (Pino JSON) for every admin action: approve/reject/suspend
- Separate `audit_logs` table — no chat content, only actions (compliance-friendly even without data retention)

## 10. Scalability Strategy

### Phase 1 — Testing / Pre-Launch (0–1,000 users)
- Single NestJS Docker container (2 vCPU / 4 GB)
- Managed Postgres (smallest tier, e.g., DO Managed PG 1 vCPU / 2 GB)
- Managed Redis (smallest tier, 512 MB)
- All on Docker Compose on a single VPS (~$40/month total)
- Cloudflare R2 (free tier covers <10 GB)
- Agora free tier: 10,000 call mins/month

### Phase 2 — Early Growth (1K–100K users)
- 2–4 NestJS instances behind NGINX (horizontal scaling)
- Socket.IO Redis adapter enables multi-node real-time
- Managed Postgres scaled up (4 vCPU / 8 GB, +read replica for reporting queries)
- Redis cluster (primary + replica)
- Agora: pay-as-you-go (~$0.99 per 1k voice min, ~$3.99 per 1k video min)
- CDN: R2 already global; Flutter assets via GitHub CDN or Cloudflare Pages

### Phase 3 — Scale (100K–1M+ users)
- **Kubernetes** (EKS/GKE/DOKS) — 10+ NestJS pods with HPA
- **PgBouncer** for connection pooling
- Postgres: primary + 2 read replicas, partition `consultations` by month
- **Separate Socket.IO cluster** if WebSocket load warrants (handoff via Redis)
- **Migration candidate**: move from Agora to self-hosted LiveKit cluster if video minute bill exceeds $5K/month (break-even analysis)
- **Observability stack**: Prometheus + Grafana + Loki
- **Edge caching** for public lawyer listings via Cloudflare Workers

### Horizontal Scaling Enablers (Built-In from Day 1)
- Stateless API — any instance serves any request
- All state in Postgres + Redis (no in-memory app state)
- Socket.IO with Redis adapter (rooms work across nodes)
- JWT verification is CPU-local (no DB lookup per request)
- Rate limiting uses Redis (consistent across instances)

## 14. Key Architectural Trade-offs

1. **Client-side chat history** vs server-stored history
   - ✅ Chose client-side — matches requirement, cheaper at scale, better privacy
   - ⚠️ Trade-off: users lose history on phone loss; no cross-device sync

2. **Separate role tables** vs unified `users` table + role column
   - ✅ Chose separate — cleaner schema, clearer permissions, field divergence significant
   - ⚠️ Trade-off: slightly more code for cross-table email uniqueness checks

3. **Agora (managed)** vs self-hosted LiveKit
   - ✅ Chose Agora for MVP — zero DevOps burden, proven reliability
   - ⚠️ Trade-off: per-minute cost; migration path built in via CallService abstraction

4. **BYTEA for license, R2 for photos** vs all-R2
   - ✅ Hybrid — license is sensitive + low-traffic, photos are hot + cacheable
   - ⚠️ Trade-off: two storage systems to manage; DB grows slowly with lawyer count

5. **NestJS + Prisma** vs bare Express / Fastify
   - ✅ Chose NestJS — module structure ideal for Cursor AI, built-in guards/interceptors
   - ⚠️ Trade-off: heavier framework, more abstraction layers to learn

6. **Socket.IO** vs raw WebSocket (ws)
   - ✅ Chose Socket.IO — auto-reconnect, rooms, Redis adapter out-of-box
   - ⚠️ Trade-off: slightly heavier wire protocol than raw WS

7. **Single Flutter binary with role-based UI** vs separate apps per role
   - ✅ Chose single binary — one codebase to maintain, admin rare enough to live alongside
   - ⚠️ Trade-off: larger APK, admin UI code ships to all users (no security risk since server-gated)

## 16. Implementation Order for Cursor AI

When Cursor AI begins implementation, the recommended build order is:

1. **Foundation** — NestJS skeleton, Prisma schema, Docker Compose with Postgres + Redis
2. **AuthModule + NotificationModule (email only)** — signup, OTP, login working end-to-end
3. **UserModule + LawyerModule** — profile CRUD, lawyer search (sans filters first)
4. **LicenseModule + AdminModule + SuperAdminModule** — lawyer approval workflow
5. **ChatModule** — Socket.IO gateway, online + offline flows, pending_messages
6. **NotificationModule (FCM)** — push for offline messages
7. **CallModule + ConsultationModule** — Agora integration, call lifecycle
8. **MediaModule** — R2 presigned uploads for profile photos
9. **RatingModule** — post-consult ratings
10. **PaymentModule (stub)** — schema + zeroed-out endpoints
11. **Flutter scaffolding** — clean architecture folders, theme, router
12. **Flutter screen-by-screen build** following Design.md

Each step has its own unit + integration tests before moving on.
